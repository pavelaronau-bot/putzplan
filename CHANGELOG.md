# CHANGELOG

Формат: Keep a Changelog. Версионирование: семантическое.

## [0.1.0] — 2026-07-20 — Sprint 1: первый вертикальный срез

### Добавлено — backend
- FastAPI на Python 3.12 с асинхронным SQLAlchemy 2 и asyncpg.
- Аутентификация: вход по e-mail, Argon2id (19 МиБ, 2 итерации), access-токен
  15 минут, refresh 30 дней в HttpOnly + Secure + SameSite=Strict cookie,
  ротация с обнаружением повторного использования, выход и отзыв всех сессий.
- Защита от перебора: 5 неудачных попыток → блокировка на 15 минут,
  ограничение частоты запросов, одинаковый ответ для неверного пароля
  и несуществующего пользователя.
- Контекст арендатора: `SET LOCAL app.company_id` внутри транзакции,
  `company_id` только из сессии, чужая запись возвращает 404.
- RBAC: централизованная зависимость `require(permission)`, deny-by-default,
  права из БД с индивидуальными grant и deny, запрет повышения своих прав,
  защита последнего активного владельца.
- Пользователи: список с пагинацией, фильтрами и сортировкой по белому списку,
  создание, изменение, деактивация с обязательной причиной, просмотр
  и отзыв сессий.
- Роли: список, создание пользовательской роли, изменение, назначение прав,
  защита системных ролей.
- Журнал: отдельный сервис и отдельная роль БД, append-only, хеш-цепочка
  `chain_seq`, очистка секретов из полезной нагрузки.
- Наблюдаемость: JSON-логи с `request_id`, счётчики, `/health` и `/ready`
  с проверкой обоих пулов и запаса партиций.
- Alembic: базовая ревизия поверх схемы Gate 1 и ревизия Sprint 1.

### Добавлено — frontend
- React 18 + TypeScript 5 + Vite 5, типизированный клиент по контракту OpenAPI.
- Промышленная страница входа без демонстрационных подсказок.
- Access-токен только в памяти вкладки, молчаливое обновление по refresh-cookie.
- Страница пользователей: таблица, поиск, фильтр статуса, пагинация,
  создание и редактирование в модальном окне, деактивация.
- Страница ролей и прав с матрицей по модулям.
- Профиль с перечнем собственных прав.
- Состояния загрузки, пустого списка, ошибки с отображением `request_id`.

### Добавлено — инфраструктура
- Docker Compose: PostgreSQL/PostGIS, Redis, отдельный шаг миграций, API, frontend.
- Dockerfile бэкенда и фронтенда с непривилегированным пользователем и healthcheck.
- Makefile: up, down, migrate, seed, test, smoke, lint, openapi, reset-db.
- GitHub Actions: качество кода, SQL-тесты, миграции, backend-тесты,
  frontend-тесты, e2e, безопасность, сборка образов.
- `.env.example` без реальных секретов.

### Добавлено — тесты
- 26 модульных: пароли, токены, ротация, вычисление прав, инварианты ролей,
  очистка журнала, ограничение частоты.
- 24 интеграционных на живой PostgreSQL: вход, ротация, отзыв, изоляция
  арендаторов, утечка контекста через пул, RLS под непривилегированной ролью,
  жизненный цикл пользователя, роли, журнал, пагинация, инъекции.
- 43 smoke-проверки вертикального среза по HTTP.
- 7 тестов фронтенда, сценарий Playwright.

### Исправлено
- Отзыв цепочки сессий откатывался вместе с исключением при обнаружении
  повторного использования refresh-токена.
- Повреждённый хеш пароля вызывал ошибку вместо отказа во входе.
- Создание пользователя падало из-за невыводимого типа параметра в asyncpg.
- Сид не обновлял e-mail при конфликте по идентификатору.
- Кнопка входа блокировалась без объяснения причины.

### Известные ограничения
- Playwright и Docker Compose не запускались в среде разработки.
- Счётчик ограничения частоты хранится в памяти процесса.
- Права кэшируются на 60 секунд.
- Реализован один вертикальный срез; бизнес-модули — этапы 1–5 дорожной карты.
