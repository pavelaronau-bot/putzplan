# Пошаговая инструкция: запуск проверки в GitHub Actions

Для владельца репозитория. Занимает около 15 минут, из них большая часть —
ожидание пайплайна. Ничего в коде менять не нужно.

---

## Шаг 1. Создать репозиторий на GitHub

1. Откройте <https://github.com/new>.
2. Имя, например, `putzplan`. Видимость — приватная или публичная, обе подходят.
3. **Не добавляйте** README, .gitignore и лицензию: репозиторий должен быть пустым.
4. Нажмите *Create repository*.

## Шаг 2. Загрузить код

Распакуйте архив и выполните в каталоге `putzplan`:

```bash
git init                       # если каталог ещё не репозиторий
git add -A
git commit -m "PUTZPLAN Sprint 1"
git branch -M main
git remote add origin https://github.com/<владелец>/<репозиторий>.git
git push -u origin main
```

Если архив уже содержит историю git, достаточно:

```bash
git remote add origin https://github.com/<владелец>/<репозиторий>.git
git push -u origin main
git push origin feature/sprint-2      # ветка Sprint 2, в main не сливать
```

Проверьте на GitHub: в корне видны каталоги `backend`, `frontend`,
`infrastructure` и файл `.github/workflows/ci.yml`.

## Шаг 3. Добавить Secrets (необязательно)

Пайплайн запускается **без единого секрета** — у всех паролей есть значения
по умолчанию для CI. Если политика организации требует явных секретов:

1. Repository → **Settings** → **Secrets and variables** → **Actions**.
2. Вкладка **Secrets** → *New repository secret*. Добавьте четыре:

| Name | Значение (любое, только латиница и цифры) |
|---|---|
| `CI_DB_MIGRATION_PASSWORD` | например, `ci_migration_2026` |
| `CI_DB_RUNTIME_PASSWORD` | например, `ci_runtime_2026` |
| `CI_DB_AUDIT_PASSWORD` | например, `ci_audit_2026` |
| `CI_DB_READONLY_PASSWORD` | например, `ci_readonly_2026` |

Это одноразовые пароли тестовой базы внутри раннера. Production-пароли сюда
вносить нельзя.

3. Вкладка **Variables** → *New repository variable*, **только для первого прогона**:

| Name | Value |
|---|---|
| `ALLOW_MISSING_OPENAPI_BASELINE` | `true` |

Она нужна, потому что при первой загрузке в базовой ветке ещё нет файла
`openapi/openapi.json` для сравнения. **После первого зелёного прогона переменную
удалите** — иначе проверка совместимости контракта останется ослабленной.

## Шаг 4. Открыть Actions и запустить проверку

1. Перейдите на вкладку **Actions** репозитория.
2. При первом заходе GitHub спросит подтверждение — нажмите
   *I understand my workflows, go ahead and enable them*.
3. В списке слева выберите workflow **CI**.
4. Справа нажмите **Run workflow**.
5. Ветка — `main`. Поле *Разрешить отсутствие базового контракта OpenAPI*
   оставьте `false`, если добавили переменную из шага 3, иначе поставьте `true`.
6. Нажмите зелёную кнопку **Run workflow**.

Прогон стартует за несколько секунд. Полный цикл занимает примерно 12–20 минут:
дольше всего идут задачи `docker` (сборка образов) и `e2e` (установка Chromium).

## Шаг 5. Посмотреть результат

1. Actions → верхняя строка списка — ваш прогон.
2. Откройте его: слева список из восьми задач.
3. Дождитесь завершения всех. Ожидаемая картина:

```
✅ backend-quality    ruff и mypy
✅ database           56 SQL-тестов + структурная валидация
✅ backend-tests      92 теста, миграции, контракт OpenAPI
✅ frontend           9 тестов, сборка, npm audit
✅ e2e                smoke 45 проверок + Playwright Chromium
✅ security           bandit, pip-audit, секреты, TODO
✅ docker             сборка, up, /health, /ready, smoke, down -v, повторный запуск
✅ release-gate       все обязательные задачи успешны
```

4. Если какая-то задача красная — откройте её и разверните упавший шаг.
   В задачах `docker` и `e2e` при падении автоматически выгружаются логи
   контейнеров и отчёт Playwright.

## Шаг 6. Что передать для аудита

Скопируйте и пришлите:

1. **Ссылку на прогон** целиком:
   `https://github.com/<владелец>/<репозиторий>/actions/runs/<RUN_ID>`
2. **Run ID** — число в конце этой ссылки.
3. **Commit SHA** прогона — должен быть `3eacb31…`
4. Ссылки на три ключевые задачи: `docker`, `e2e`, `release-gate`
   (откройте задачу и скопируйте адрес из строки браузера).
5. Архив логов: на странице прогона справа вверху ⋯ → *Download log archive*.
6. Артефакты внизу страницы прогона: `pytest-report`, при наличии `playwright-report`.

## Частые затруднения

| Симптом | Причина и решение |
|---|---|
| Вкладка Actions пуста | workflow не включён: нажмите *Enable workflows* |
| Задача `docker` падает на `docker compose up` | посмотрите шаг *Логи при ошибке* — он выгружает вывод контейнеров |
| `e2e` падает на установке Chromium | повторите запуск: иногда подводит сеть раннера |
| `backend-tests` падает на проверке контракта | выполните локально `make openapi` и закоммитьте `openapi/` |
| `release-gate` красный при зелёных задачах | значит какая-то задача пропущена или отменена — проверьте список |

## Важное

Пока прогон не станет зелёным, статусы **Docker**, **Playwright** и **Sprint 1**
остаются **NOT READY**. Ветку `feature/sprint-2` в `main` не сливайте:
разработка Sprint 2 приостановлена до закрытия Sprint 1.
