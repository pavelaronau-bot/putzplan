# DEVELOPMENT_ROADMAP — план разработки

Отсчёт от точки «Architecture v1.0 — Approved for Development».
Оценки — для команды из трёх разработчиков (backend, frontend, mobile)
и одного инженера инфраструктуры на частичной занятости.

## Определение готовности (Definition of Done)

Задача считается выполненной, когда:

1. Код прошёл ревью и слит в `main`.
2. Изменения схемы оформлены миграцией и покрыты тестом в `db/tests/`.
3. Эндпоинты описаны в `openapi.yaml`, права заведены в `permissions`.
4. Автотесты зелёные: тесты базы, тесты API, проверки безопасности в CI.
5. Каждое изменяющее действие пишет событие в журнал.
6. Отказ в доступе проверен негативным тестом.
7. Документация обновлена в том же PR.
8. Нет `TODO` в миграциях и архитектурных документах.

## Этап 1 — Ядро планирования (4–6 недель)

| Задача | Результат |

## Порядок и зависимости

```
Gate 1 ──▶ Этап 1 ──▶ Этап 2 ──▶ Этап 3 ──▶ Этап 4 ──▶ Этап 5 ──▶ первый платящий клиент
             │                       ▲
             └── frontend календаря ─┘ (параллельно после фиксации контракта)
```

Правила очерёдности:

1. Frontend начинается только после фиксации контракта соответствующего среза.
2. Мобильное приложение — после этапа 1: без заданий ему нечего показывать.
3. Биллинг — после этапа 3: тарифицировать нечего, пока нет продуктовой ценности.
4. Пентест — до первого платящего клиента, не после.

Суммарная оценка до первой продажи: **17–24 недели** сверх завершённого Gate 1.
