## 3. Исправление №1 — межарендаторная целостность

### Что сделано
Для каждой tenant-owned связи добавлены `UNIQUE (company_id, id)` у родителя и
составной внешний ключ `FOREIGN KEY (company_id, entity_id) REFERENCES parent (company_id, id)`.
Простые ключи сохранены как дополнение — они дают семантику `ON DELETE`.
Обоснование выбора — ADR-0001.

Покрытые связи: `objects → clients`, `client_contacts → clients`,
`jobs → objects/series`, `job_assignments → jobs/employees`,
`timesheets → jobs/employees`, `inventory → objects/materials`,
`stock_movements → objects/materials/jobs`, `incidents → objects/jobs/employees/clients`,
`photos/signatures/checklist_results → jobs`, `departments → branches`,
`employees → departments`, `users → employees/branches/departments`,
`objects → employees` (основной, резервный, руководитель), `absences → employees`,
`invoices → clients/subscriptions`, `sync_queue → users/jobs/devices`,
`secret_reveals → objects/jobs/users`, `notifications → users`,
`audit_logs → users`, `domain_events → users`, `trusted_devices → users`.

Побочная находка: `trusted_devices` не имела `company_id` — добавлена.

## 4. Исправление №2 — автоматические тесты

### Что сделано
Каркас на PL/pgSQL в схеме `tst`: `tst.ok`, `tst.is`, `tst.is_count`,
`tst.throws(name, sql, sqlstate)`, `tst.lives`, `tst.summary`, `tst.finish`.
Негативный тест **проваливается**, если запрещённая операция прошла —
визуальный просмотр вывода не требуется.

Раннер `run_tests.sh`: пересоздаёт базу, применяет миграции, выполняет
тесты под тремя разными ролями через TCP с паролями. Роль `putzplan_runtime`
никому не выдаётся — тесты подключаются как отдельные пользователи.

### Результат
```

✅ Все тесты пройдены
код возврата успешного прогона: 0
код возврата при провале теста:  1
```

Проверка кода возврата выполнена отдельно: искусственный провал
(`tst.record(..., false)`) завершает раннер кодом 1 — CI падает.

## 5. Исправление №3 — партиции

| Параметр | Значение |
|---|---|
| Партиционированные таблицы | `audit_logs`, `domain_events`, `notifications`, `login_attempts` |
| Ключ | время события, входит в первичный ключ |
| Горизонт | 18 месяцев, фактически 561 день |
| Создание | `ensure_partitions()`, идемпотентна, вызывается ежемесячным заданием |
| Мониторинг | `partition_headroom()` — алерт при остатке < 60 дней |
| DEFAULT | создаются; непустая DEFAULT — авария P2 |
| Перераспределение | `redistribute_default_partition(table)` |
| Ретеншн | `detach_expired_partitions(dry_run)`, сроки в реестре `partitioned_tables` |

Решение по DEFAULT-партиции принято явно и обосновано в ADR-0004:
потеря записи журнала хуже, чем строка в неоптимальной партиции.

Тесты: вставка в текущую партицию, вставка на 12 месяцев вперёд,
пустота DEFAULT, известность горизонта, запас не менее 500 дней.
Подробности — `PARTITIONING_AND_RETENTION.md`.

## 6. Исправление №4 — детерминированная цепочка

| Было | Стало |
|---|---|
| Порядок по `server_time` | Монотонный `chain_seq` внутри компании |
| Только `prev_hash` | `prev_id` + `prev_hash` + `row_hash` |
| Время входило в хеш | Время из хеша убрано |
| `audit_chain_state.last_id` не использовался | Используется и обновляется атомарно вместе с `last_seq` и `last_hash` |
| Проверка сортировала по времени | `audit_verify_chain` идёт строго по `chain_seq` |

Найденный при тестировании дефект: при многострочном `INSERT` состояние
обновлялось `AFTER`-триггером, и все строки получали `chain_seq = 1`.
Обновление перенесено в `BEFORE`-триггер (ADR-0002).

Результат теста параллельности:
```
PASS  параллельные вставки: цепочка без разрывов и ветвления
PASS  chain_seq уникален внутри компании
PASS  chain_seq непрерывен: max = количество записей
PASS  каждая запись ссылается на предыдущую по prev_id
PASS  состояние цепочки совпадает с фактическим максимумом
```

Подробности, регламент контроля и порядок реакции на повреждение —
`AUDIT_CHAIN_DESIGN.md`.

Реализован и проверен: аутентификация, контекст арендатора,

| Слой | Результат | Проверка |
|---|---|---|

| Backend | login / refresh / logout / password change, контекст арендатора, RBAC middleware | deny-by-default и журнал |

| Security | argon2id, ротация refresh с обнаружением повтора, лимиты, блокировка | негативные тесты |

Найденные при тестировании дефекты бэкенда: запись журнала падала на
`RETURNING` из-за RLS (добавлен контекст арендатора для пула журнала) и
`bigint` возвращался строкой, из-за чего `chain_seq` сравнивался
лексикографически (ADR-0006).

Frontend production-логина в этот срез не входит: он начинается после
фиксации контракта, согласно запрету на массовую frontend-разработку до Gate 1.
