# PARTITIONING_AND_RETENTION — партиционирование и сроки хранения

Проверено на PostgreSQL 16.14. Все функции существуют в схеме и покрыты тестами.

## 1. Что партиционируется

| Таблица | Ключ | Период | Причина |
|---|---|---|---|
| `audit_logs` | `server_time` | месяц | растёт быстрее всех, хранится 3 года |
| `domain_events` | `recorded_at` | месяц | источник уведомлений и веб-хуков |
| `notifications` | `created_at` | месяц | массовые записи, короткий срок жизни |
| `login_attempts` | `created_at` | месяц | защита от перебора, анализ инцидентов |

Реестр хранится в таблице `partitioned_tables` — функции обслуживания берут
список оттуда, а не из жёстко зашитого перечня.

```sql
SELECT * FROM partitioned_tables;
--  table_name     | part_column | retention_months | horizon_months
--  audit_logs     | server_time |               36 |             18
--  domain_events  | recorded_at |               12 |             18
--  notifications  | created_at  |                6 |             18
--  login_attempts | created_at  |               12 |             18
```

Ключ партиционирования входит в первичный ключ: `PRIMARY KEY (id, server_time)`.

## 2. Горизонт: 18 месяцев

Партиции создаются на 18 месяцев вперёд функцией `ensure_partitions()`.
При развёртывании создаётся 19 партиций на таблицу (текущий месяц + 18) плюс
DEFAULT-партиция. Фактически в проверенной базе: **76 партиций**, запас
**561 день**.

```sql
SELECT * FROM ensure_partitions();      -- горизонт по умолчанию из реестра
SELECT * FROM ensure_partitions(24);    -- вручную на 24 месяца
```

Функция идемпотентна: повторный вызов не создаёт дублей и не мешает работе.

## 3. Планировщик

| Задание | Частота | Что делает |
|---|---|---|
| `ensure_partitions()` | 1-го числа каждого месяца, 02:10 | создаёт недостающие партиции |
| `partition_headroom()` | каждый час | метрика запаса для мониторинга |
| `partition_default_rows()` | каждый час | контроль, что DEFAULT пуста |
| `detach_expired_partitions(false)` | 1-го числа, 03:30 | отсоединяет партиции сверх срока хранения |

Вариант с pg_cron:

```sql
SELECT cron.schedule('partitions-ensure', '10 2 1 * *',
                     $$SELECT ensure_partitions()$$);
SELECT cron.schedule('partitions-detach', '30 3 1 * *',
                     $$SELECT detach_expired_partitions(false)$$);
```

Вариант без pg_cron — Kubernetes CronJob:

```yaml
schedule: "10 2 1 * *"
command: ["psql", "-v", "ON_ERROR_STOP=1", "-c", "SELECT ensure_partitions()"]
```

Задание выполняется ролью `putzplan_migration`: рабочей роли право
`EXECUTE` на `ensure_partitions` не выдано.

## 4. Мониторинг

```sql
SELECT * FROM partition_headroom();
--  table_name | last_partition      | covered_until | days_left | alert
--  audit_logs | audit_logs_2028_01  | 2028-02-01    |       561 | false
```

| Проверка | Порог | Приоритет |
|---|---|---|
| `days_left < 60` у любой таблицы | предупреждение | P2 |
| `days_left < 30` | ошибка | P1 |
| `partition_default_rows() > 0` | строки попали в DEFAULT | P2 |
| Функция `ensure_partitions` не выполнялась 40 дней | пропуск задания | P2 |

Проверка `days_left` выведена и в `/readyz` приложения — деградация видна
сразу на дашборде готовности.

## 5. Решение по DEFAULT-партиции

**Решение: DEFAULT-партиции создаются.** Обоснование: при пропуске планового
задания вставка в журнал не должна падать — потеря записи журнала хуже,
чем строка в неоптимальной партиции. При этом непустая DEFAULT считается
аварией и имеет процедуру разбора.

Известное ограничение: пока DEFAULT содержит строки, попадающие в диапазон
новой партиции, команда `ATTACH PARTITION` требует полного сканирования и
может завершиться ошибкой. Поэтому предусмотрена процедура перераспределения:

```sql
CALL redistribute_default_partition('audit_logs');
-- 1. отсоединяет DEFAULT
-- 2. создаёт недостающие партиции на 24 месяца
-- 3. переносит строки обратно в родителя (они попадают в нужные партиции)
-- 4. присоединяет DEFAULT обратно
```

Порядок действий при срабатывании алерта «DEFAULT не пуста»:

1. Зафиксировать инцидент, определить причину пропуска задания.
2. Выполнить `SELECT ensure_partitions(24)`.
3. Выполнить `CALL redistribute_default_partition('<таблица>')` в окне низкой нагрузки.
4. Убедиться: `partition_default_rows()` возвращает 0.
5. Восстановить расписание задания и проверить его выполнение вручную.

## 6. Сроки хранения и архивирование

| Таблица | Хранение | По истечении |
|---|---|---|
| `audit_logs` | 36 месяцев | отсоединение → WORM-хранилище, удаление не ранее 3 лет |
| `domain_events` | 12 месяцев | отсоединение → холодное хранилище Parquet |
| `notifications` | 6 месяцев | удаление |
| `login_attempts` | 12 месяцев | удаление |

```sql
SELECT * FROM detach_expired_partitions(true);   -- предварительный просмотр
SELECT * FROM detach_expired_partitions(false);  -- фактическое отсоединение
```

Отсоединение не удаляет данные: таблица остаётся в базе как самостоятельная и
может быть выгружена, заархивирована и затем удалена вручную. Для журнала
удаление партиции требует подтверждения владельца платформы и фиксируется
в журнале платформы.

Порядок архивирования журнала:

```bash
# 1. Отсоединить
psql -c "SELECT detach_expired_partitions(false)"
# 2. Выгрузить в Parquet и положить в WORM-бакет
pg_dump -t audit_logs_2023_07 --data-only | zstd > audit_logs_2023_07.sql.zst
aws s3 cp audit_logs_2023_07.sql.zst s3://putzplan-archive/audit/ --object-lock-mode COMPLIANCE
# 3. Проверить контрольную сумму и только затем удалить таблицу
psql -c "DROP TABLE audit_logs_2023_07"
```

## 7. Восстановление партиции из архива

1. Загрузить файл из архива, проверить контрольную сумму.
2. Восстановить как отдельную таблицу `audit_logs_2023_07_restored`.
3. Проверить диапазон дат и целостность хеш-цепочки на этом отрезке.
4. Присоединить: `ALTER TABLE audit_logs ATTACH PARTITION audit_logs_2023_07_restored FOR VALUES FROM (…) TO (…)`.
5. Зафиксировать восстановление в журнале платформы.
