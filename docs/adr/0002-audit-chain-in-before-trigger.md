# ADR-0002. Состояние хеш-цепочки обновляется в BEFORE-триггере

**Дата:** 20.07.2026 · **Статус:** принято

## Контекст
Первая реализация обновляла `audit_chain_state` в `AFTER INSERT` триггере,
чтобы иметь готовый `id` строки. При многострочном `INSERT` строковые
`AFTER`-триггеры выполняются после завершения оператора, поэтому все строки
читали одно состояние и получали `chain_seq = 1`. Ошибку нашёл автоматический тест.

## Решение
Расчёт хеша и обновление состояния выполняются в одном `BEFORE INSERT`
триггере под `pg_advisory_xact_lock` по компании. Значение `id` доступно
уже в `BEFORE`, так как значения по умолчанию, включая `IDENTITY`,
вычисляются до срабатывания триггера.

## Последствия
- Многострочные вставки получают корректную последовательность.
- Порядок цепочки определяется `chain_seq`, а не `server_time`.
- Параллельные вставки сериализуются по компании: пропускная способность
  журнала ограничена одной транзакцией на компанию единовременно — приемлемо
  при ожидаемых объёмах.
