# ADR-0001. Составные внешние ключи для межарендаторной целостности

**Дата:** 20.07.2026 · **Статус:** принято

## Контекст
Простой внешний ключ по `id` гарантирует существование записи, но не то, что
она принадлежит той же компании. RLS защищает чтение и запись в рамках сессии,
но ошибка в коде, выполняющая вставку под корректным `company_id` со ссылкой
на чужую сущность, проходит обе проверки.

## Решение
Для каждой tenant-owned связи добавлены `UNIQUE (company_id, id)` у родителя и
составной внешний ключ `FOREIGN KEY (company_id, entity_id) REFERENCES parent (company_id, id)`.
Простые ключи сохранены как дополнение: они дают понятную семантику `ON DELETE`
и явные сообщения об ошибке.

## Последствия
- Ссылка на чужого арендатора невозможна физически: 46 объявленных ограничений, 107 с учётом партиций.
- Требуется дополнительный уникальный индекс на каждом родителе — расход места и небольшая цена при вставке.
- `trusted_devices` получила колонку `company_id`, ранее принадлежность выводилась через пользователя.
- RLS сохраняется как второй независимый слой.
