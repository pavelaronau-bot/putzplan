# Разбор падения CI на шаге pip-audit

## Симптом

Задача `security`, шаг «Уязвимости зависимостей Python» завершался кодом 1.
В логе — только успешная установка пакетов и `Process completed with exit code 1`.
Вывода самого `pip-audit` не было.

## Причина

Найдены четыре независимых дефекта; первый объясняет симптом полностью.

### 1. Шаг падал до запуска pip-audit

```bash
set -euo pipefail
IGNORES=$(grep -vE '^#|^$' security/pip-audit-allowlist.txt | awk '...')
```

Файл исключений состоит только из комментариев. `grep` без совпадений
возвращает код 1, `pipefail` передаёт его подстановке команд, `set -e`
завершает шаг. До `pip-audit` выполнение не доходило — отсюда пустой лог.

Воспроизведение:

```
$ bash /tmp/repro.sh
--- шаг начался ---
код возврата: 1
```

### 2. `--strict` спотыкается о собственный пакет проекта

```
ERROR:pip_audit._cli:putzplan-backend: Dependency not found on PyPI
and could not be audited: putzplan-backend (0.1.0)
```

В строгом режиме любая зависимость, которую не удалось сопоставить с PyPI,
считается ошибкой. Пакет `putzplan-backend` на PyPI отсутствует по определению.

### 3. Аудитировалось всё окружение, а не зависимости проекта

Запуск `pip-audit` без файла требований проверяет установленные пакеты
целиком, включая системные. В контейнере это дало **128 уязвимостей
в 18 пакетах** (`mistune`, `lxml`, `httplib2` и прочие), не имеющих
отношения к продукту. Результат зависел от образа раннера и был
невоспроизводим.

### 4. Пакет невозможно установить неявляемым способом

```
error: Multiple top-level packages discovered in a flat-layout: ['app', 'alembic'].
```

В `pyproject.toml` отсутствовала секция `[build-system]` и настройка
обнаружения пакетов, поэтому обычная установка завершалась ошибкой —
работал только режим `-e`.

## Исправления

| Дефект | Решение |
|---|---|
| 1 | Разбор allowlist построчно в цикле `while read`, без `grep`. Файл из одних комментариев больше не роняет шаг |
| 2 | Аудит выполняется по списку зависимостей (`pip freeze` без собственного пакета), а не по установленным дистрибутивам |
| 3 | Аудит идёт в изолированном venv, где установлены только зависимости проекта |
| 4 | В `pyproject.toml` добавлены `[build-system]` и `[tool.setuptools.packages.find]` с `include = ["app*"]` |

Логика вынесена в `infrastructure/scripts/audit_python_deps.sh`.
Скрипт запускают и CI, и разработчик командой `make security`, поэтому
расхождение «локально работает, в CI падает» исключено.

## Найденная уязвимость и её устранение

После исправления проверка заработала и **нашла настоящую проблему**:

```
Found 9 known vulnerabilities in 1 package
starlette 0.46.2  PYSEC-2026-161, PYSEC-2026-248, PYSEC-2026-249,
                  PYSEC-2026-1941, PYSEC-2026-1942, PYSEC-2026-2280,
                  PYSEC-2026-2281
```

`starlette` приходил транзитивно через ограничение `fastapi>=0.115,<0.116`.
Проверка **не отключалась и в allowlist не вносилась** — обновлены версии:

```toml
"fastapi>=0.139,<0.140",
# starlette закреплён явно: версии ниже 1.3.1 содержат известные уязвимости
"starlette>=1.3.1",
```

Результат: `fastapi 0.139.2`, `starlette 1.3.1`, аудит — `No known vulnerabilities found`.

## Проверка после исправления

| Проверка | Результат |
|---|---|
| `bash infrastructure/scripts/audit_python_deps.sh` | код 0, уязвимостей нет |
| Тот же скрипт с уязвимой `starlette==0.46.2` | код 1, 9 уязвимостей — гейт срабатывает |
| Allowlist с записью `PYSEC-2026-9999` | «исключений из allowlist: 1», разбор корректен |
| Allowlist из одних комментариев | «исключений: 0», шаг не падает |
| Backend-тесты на FastAPI 0.139 и Starlette 1.3 | 92 passed |
| Smoke вертикального среза | 45 passed |
| SQL-тесты, Vitest, сборка фронтенда | 56 / 9 / успешно |
| mypy, ruff, bandit | без замечаний, 0 high |
| Контракт OpenAPI | изменения аддитивные, гейт совместимости пройден |

Обновление FastAPI до 0.139 и Starlette до 1.3 обратной совместимости
не нарушило: контракт пополнился только полями описания ошибок валидации.
