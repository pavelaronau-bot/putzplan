## 1. Критический дефект: победитель гонки терял токены

### Что было
Функция `auth_rotate_session` блокировала строку через `SELECT … FOR UPDATE`.
Проигравший запрос дожидался блокировки, видел старую сессию уже отозванной
и трактовал это как повторное использование украденного токена. Дальше он
отзывал всё семейство — вместе с сессией, которую только что создал победитель.

Дефект воспроизведён до исправления:

```
победителей: 1 | коды: [200, 409 × 9]
токен победителя после гонки: 401 session_revoked
```

Прежний тест проверял только коды 409 у проигравших, поэтому дефект не ловил.
Это ровно тот случай, который запрещён разделом 6 задания.

### Что сделано
Миграция `0004` вводит явное состояние замены вместо трактовки любой
отозванной сессии как кражи:

| Колонка | Назначение |
|---|---|
| `rotated_at` | момент замены токена |
| `replaced_by_session_id` | ссылка на сессию-преемника |

Логика `auth_rotate_session` (grace-window по умолчанию 30 секунд):

| Состояние старой сессии | Результат | Действие |
|---|---|---|
| активна | `rotated` | создаётся преемник, старая помечается `rotated` |
| отозвана как `rotated`, в пределах окна | `race` | **ничего не отзывается**, событие `REFRESH_RACE_DETECTED` |
| отозвана как `rotated`, вне окна | `reuse` | отзыв всего семейства, событие `REFRESH_REUSE_DETECTED` |
| отозвана по другой причине | `reuse` | отзыв всего семейства |
| истекла | `expired` | 401 |
| пользователь неактивен | `inactive_user` | 403 |
| токен не найден | `not_found` | 401 |

Состояние `race_lost` **удалено**: под блокировкой строки условный
`UPDATE … WHERE revoked_at IS NULL` не может вернуть ноль строк, то есть путь
был недостижим. Отсутствие проверяется тестом `test_rotation_states_are_exhaustive`.

Новый raw refresh-токен в базе не хранится: сохраняется только SHA-256 хеш,
как и раньше. Идемпотентный повтор ответа не реализован намеренно — выбран
вариант «отдельное состояние ротации», прямо разрешённый заданием.

## 3. Release Gate: устранены fail-open места

| Было | Стало |
|---|---|
| `git fetch … \|\| exit 0` в проверке контракта | `set -euo pipefail`, недоступность ветки — ошибка |
| `git show … \|\| exit 0` при отсутствии baseline | падение с объяснением; пропуск только при явной переменной `ALLOW_MISSING_OPENAPI_BASELINE=true` |

**e2e**, security, **docker**.
